# AGENT.md

## 1. 文档目的

本文件用于指导未来的大模型或自动化编码代理在 `zero-caption` 仓库内开展工作。

目标只有三个：

1. 先理解当前骨架，再扩展功能，不凭设计稿想当然改结构。
2. 严格遵守分层边界，避免把业务、UI、基础设施揉在一起。
3. 围绕 MVP 主链路推进：`导入 -> 识别 -> 翻译 -> 导出`。

---

## 2. 产品定位

`zero-caption` 是一款面向 Windows 的本地优先视频字幕生成与翻译桌面工具。

核心原则：

- UI 使用 `PySide6`
- 主体是单机桌面应用，不做传统前后端分离
- 音视频处理、本地识别、缓存、导出尽量本地完成
- 云端仅承担字幕文本翻译，不上传原始视频和音频

设计依据来自：

- `design/视频字幕生成与翻译软件_应用框架设计.md`
- `design/推荐项目骨架说明.md`

## 4. 目录与职责

### `app/`

应用编排与装配层。

职责：

- 启动 Qt 应用
- 加载配置
- 初始化日志、工作区、外部适配器
- 组装 `MainWindow` 与业务服务

约束：

- 不写具体业务规则
- 不直接放 FFmpeg、翻译、数据库等细节逻辑

### `ui/`

PySide6 界面层。

职责：

- 展示页面、对话框、导航、状态
- 收集用户输入
- 触发服务调用
- 订阅任务进度和状态变化

约束：

- 不做重计算
- 不直接调用 FFmpeg、ASR、翻译 HTTP API
- 不在页面类里堆积业务编排

### `core/`

业务核心层。

职责：

- 定义实体、枚举、DTO、抽象端口
- 编排核心用例
- 维护任务状态机和处理流程

约束：

- 不依赖 Qt
- 不依赖具体基础设施实现
- 优先依赖 `ports/` 抽象

### `infrastructure/`

外部依赖实现层。

职责：

- 媒体处理适配器
- 翻译适配器
- 存储实现
- 任务执行实现
- 日志与工作区支持

约束：

- 负责“如何做”，不负责“是否做、按什么业务顺序做”

### `config/`

配置层。

职责：

- 默认配置
- 用户配置
- 运行参数装载

### `tests/`

测试层。

职责：

- 验证配置、目录、枚举、用例、适配器行为

## 5. 必须遵守的架构规则

1. UI 只负责交互，不负责主流程编排。
2. 主流程必须放在 `core/`，或由 `core/` 驱动的服务中。
3. 所有可替换能力都先定义端口，再写基础设施实现。
4. 外部系统调用要集中在 `infrastructure/`，不要散落在页面和窗口中。
5. 大文件处理必须流式或分段，禁止把整段视频/音频长期驻留内存。
6. 面向未来能力时，默认需要任务状态持久化、可恢复、可重试。
7. 云端翻译只传字幕文本及必要上下文，不上传原始媒体文件。
8. 新增模块优先贴近现有目录规划，不另起平行体系。

## 7. 当前代码的扩展建议

### 7.1 `app/container.py`

未来所有服务实例和适配器实例都应从这里组装。

不要：

- 在 UI 页面内部 new 出翻译器、FFmpeg 适配器、数据库连接
- 在 `main.py` 直接堆装配逻辑

### 7.2 `core/services/task_service.py`

这是当前最接近“应用服务”的入口，但现在还是占位。

建议后续把它演进为：

- 接收 UI 指令
- 创建任务
- 调用具体用例
- 汇总进度
- 更新状态

如果后面新增 `usecases/`，应让 `TaskService` 成为用例编排入口，而不是继续膨胀成全能类。

### 7.3 `infrastructure/task/job_queue.py`

当前是内存 `deque` 占位实现。

未来应演进为：

- 持久化任务记录
- 恢复未完成任务
- 支持重试和并发控制
- 对 UI 提供可订阅的状态变化

### 7.4 `ui/pages/`

当前页面是占位页。

后续页面实现要遵循：

- 页面类只组装控件与交互
- 业务动作调用服务
- 页面不要保存复杂业务状态副本
- 可恢复状态优先来自 DTO / Repository / Service

## 8. 数据与工作区约定

当前默认工作区根目录来自 `config/default.toml` 中的 `workspace_root`，默认是 `data/`。

现有目录管理由 `WorkspaceManager` 负责，当前已创建：

- `data/projects`
- `data/cache`
- `data/exports`
- `logs`

按设计文档，后续项目级目录建议向下扩展为：

```text
project_id/
  source/
  temp/
  cache/
  subtitles/
  exports/
  logs/
```

规则：

- 临时产物放 `temp/`
- 可复用中间产物放 `cache/`
- 正式字幕放 `subtitles/`
- 导出成品放 `exports/`
- 项目日志按项目隔离

## 9. 任务、状态与恢复原则

当前已有：

- `ProjectStatus`
- `TaskStatus`
- `ExportMode`

未来扩展时，优先兼容现有枚举语义，而不是频繁重命名。

设计目标要求任务具备：

- 后台执行
- 状态持久化
- 三层进度表达：总进度 / 当前步骤 / 当前文件块
- 可重试
- 可从最后成功步骤恢复

推荐的步骤级检查点：

1. 已导入
2. 已抽音频
3. 已识别
4. 已翻译
5. 已合成
6. 已导出

## 10. UI 实现原则

1. 默认保持 Windows 桌面工具风格，重信息密度和任务反馈。
2. 主窗口继续沿用“导航 + 页面栈 + 状态栏”的骨架。
3. 导入、识别、翻译、导出都应是显式动作，不做隐式后台魔法。
4. 长任务必须有进度、日志、错误信息和可恢复提示。
5. UI 线程只做交互；计算和 I/O 放后台。
6. Qt 页面间共享状态时，优先通过服务、DTO、信号/槽传递，不做全局变量拼接。

## 11. 基础设施实现原则

### 媒体处理

- 优先封装 `FFmpeg` / `ffprobe`
- 保留命令、退出码、错误输出，便于诊断
- 不在多个模块重复拼接命令行

### 翻译

- 翻译能力通过统一端口访问
- 先支持最小批量字幕翻译
- 要考虑限流、重试、上下文补充

### 存储

- 设计目标是 `SQLite + 本地项目目录`
- 结构化数据入库
- 大文件与中间产物落盘，不放数据库

### 日志

- 保留应用级日志
- 未来增加项目级日志与任务级摘要
- 关键外部调用要记录必要上下文，但避免泄露敏感密钥

## 13. 代码风格约定

1. 优先沿用现有 dataclass、显式依赖注入、清晰命名风格。
2. 新能力优先小步补全，不做跨层大重构。
3. 能用标准库先用标准库。
4. 接口签名先稳定，再扩散调用面。
5. 占位实现若升级为真实实现，要同步更新测试和文档。

### 13.1 注释总体要求

本项目面向 Python 初学者维护，默认注释策略应比常规工程更详细。

未来代理在编写或修改代码时，应把注释当作代码的一部分一起交付，而不是最后随手补几句。

要求：

1. 注释要帮助阅读者理解“这段代码为什么存在、输入输出是什么、执行顺序是什么、为什么这样写”。
2. 注释要尽量有教学作用，尤其是对 Python、分层架构、Qt 信号/槽、依赖注入、dataclass、路径处理、任务状态这些概念。
3. 不能只写结论，要在关键位置补充最小必要背景。
4. 注释、docstring、示例注释、测试注释的正文必须全部使用中文，不允许中英混写，也不允许只保留英文模板。
5. 如果必须引用代码标识符、第三方库名、协议名或类名，可以保留其原始写法，但应使用反引号包裹，并把周围解释文字写成中文。
6. 注释要和代码严格同步，代码改了，注释也必须一起改。

### 13.2 什么时候必须写注释

以下情况默认必须写注释：

1. 模块文件开头：说明这个文件属于哪一层、职责是什么、通常不应该放什么逻辑。
2. 类定义前：说明这个类解决什么问题、会被谁调用、核心状态是什么。
3. 公共方法前：说明用途、主要参数、返回值、可能的副作用。
4. 处理流程较长的方法内部：按步骤写分段注释。
5. 对新手不直观的 Python 语法或标准库用法：例如 `dataclass`、`Path`、上下文管理器、枚举、类型注解、生成器、信号/槽连接。
6. 存在架构边界的地方：解释为什么这里不能直接访问 UI、数据库或外部服务。
7. 容易误改的逻辑：例如状态机、路径约定、恢复点、缓存命中条件、任务重试规则。
8. 临时兼容写法、折中方案、已知限制：解释原因和后续替换方向。

以下情况可以少写或不写：

1. 变量名和函数名已经足够清晰的简单赋值。
2. 明显的 UI 控件创建代码，且没有额外业务含义。
3. 单行就能直接读懂的纯数据映射。

原则：宁可多解释“原因和约束”，也不要堆积“把变量赋值给变量”这类无效注释。

### 13.3 推荐的注释层次

未来新增代码时，优先按下面四层补注释：

1. 文件级注释  
   说明模块职责、所在分层、依赖边界。
2. 类级注释  
   说明类的角色、核心协作者、适用场景。
3. 方法级注释  
   说明输入、输出、执行结果、副作用、异常情况。
4. 代码块注释  
   说明当前这一段的步骤目的、关键判断和顺序原因。

如果一段逻辑对新手来说“不容易一下读懂”，就不要只靠命名硬撑，应主动补代码块注释。

### 13.4 推荐写法

推荐注释聚焦下面几类信息：

- 目的：这段代码要完成什么
- 背景：为什么要这样做
- 边界：这段代码不负责什么
- 顺序：为什么步骤不能交换
- 风险：哪里容易出错
- 结果：执行后会留下什么状态或产物

推荐风格示例：

```python
# `TaskService` 位于核心层，负责把界面层发起的动作转成具体用例调用。
# 这里不直接写 `FFmpeg` 或数据库细节，避免业务层和基础设施层耦合。
@dataclass(slots=True)
class TaskService:
    ...
```

```python
def load_settings(path: str | Path | None = None) -> Settings:
    # 如果调用方没有传配置路径，就回退到仓库内的默认配置文件。
    # 这样应用在首次启动时即使没有用户配置，也能正常运行。
    config_path = Path(path) if path else Path("config/default.toml")
```

```python
# 先确保根目录存在，再创建子目录。
# 这里拆成显式步骤，是为了让初学者更容易理解工作区结构，也便于后续扩展项目级目录。
self.root.mkdir(parents=True, exist_ok=True)
```

不推荐：

```python
# 创建路径
config_path = Path(path)

# 设置标题
self.setWindowTitle(settings.app_name)
```

上面这种注释只重复代码表面含义，没有学习价值。

### 13.5 文档字符串要求

对于下面这些代码，优先使用 docstring，而不只是行内注释：

1. 公共函数
2. 公共类
3. 后续会被多个模块调用的服务方法
4. 端口接口和适配器实现

docstring 建议至少包含：

- 这个对象的职责
- 参数含义
- 返回值含义
- 是否会修改状态、写文件、访问外部系统

docstring 同样必须全部使用中文。

推荐风格：

```python
def ensure_structure(self) -> None:
    """创建应用运行所需的工作区目录结构。

    当前骨架版本只会创建共享的顶层目录。
    后续版本可以在这里继续扩展项目级目录，
    例如 temp、subtitles 和 logs。
    """
```

如果函数很短，但它承担的是“稳定接口”角色，仍然建议写 docstring。

### 13.6 面向新手的特别要求

因为项目维护者是 Python 新手，未来代理在写代码注释时要额外注意：

1. 首次引入某个 Python 特性时，要简短解释它是什么、为什么适合这里。
2. 首次引入第三方库中的核心概念时，要解释它在项目里的角色。
3. 不要默认读者已经理解分层架构、依赖注入、DTO、Port、Adapter、Repository、Signal/Slot。
4. 可以在注释中顺手解释“小语法”，但不要写成长篇教程。
5. 遇到容易混淆的名称时，要解释命名原因。

例如：

- `@dataclass(slots=True)` 要说明 dataclass 帮助减少样板代码，`slots=True` 用于限制实例属性并减少误写属性名的风险。
- `Path` 要说明它比手写字符串路径更稳妥，适合跨平台路径拼接。
- `Enum` 要说明它用于约束可选状态，避免到处散落魔法字符串。

### 13.7 复杂流程的注释要求

对于以下类型的逻辑，必须使用“分步骤注释”：

- 任务创建与状态迁移
- 音视频处理链路
- 翻译请求组装
- 缓存命中判断
- 导出流程
- 恢复与重试逻辑

建议格式：

```python
# 第一步：校验输入，并判断是否可以复用缓存产物。
# 第二步：在重计算开始前创建或恢复任务记录。
# 第三步：把高成本工作分发给基础设施适配器。
```

如果一段逻辑跨越 15 到 30 行以上，没有任何分段注释，通常说明可读性不足。

### 13.8 测试代码的注释要求

测试也要写注释，尤其要说明“这个测试在保护什么行为”。

要求：

1. 测试名表达场景和预期。
2. 必要时在测试体内注明 arrange / act / assert 三段。
3. 如果测试数据看起来随意，要解释为什么这样构造。
4. 如果一个测试是在防止历史 bug 回归，要写出回归背景。

### 13.9 禁止事项

1. 不要用注释掩盖糟糕命名。
2. 不要写过期注释。
3. 不要把实现细节完全搬进注释，导致代码和注释重复维护。
4. 不要用“这里很复杂”“神奇逻辑”这种没有信息量的注释。
5. 不要在关键逻辑处完全无注释，要求读者自行猜测。

## 14. 代理执行清单

未来代理在开始编码前，默认先检查以下事项：

1. 这次改动属于哪一层？
2. 是否把业务逻辑错误地放进了 UI？
3. 是否应该先定义 `core/ports/` 抽象？
4. 是否需要持久化、恢复、重试或日志？
5. 是否会处理大文件，是否需要分段或流式处理？
6. 是否触碰了“云端只传字幕文本”的边界？
7. 是否需要同步补测试？

## 15. 明确不要做的事

1. 不要把仓库改造成 Web 前后端分离项目。
2. 不要让页面类直接承担 FFmpeg、翻译 API、数据库职责。
3. 不要为了一个功能平行新增第二套架构。
4. 不要在 MVP 阶段优先做复杂字幕编辑器或插件市场。
5. 不要忽略任务状态持久化与失败恢复的设计需求。
6. 不要上传原始视频或音频到云端翻译服务。

---

## 16. 一句话执行策略

先在现有骨架上，把 `导入 -> 识别 -> 翻译 -> 导出` 主链路按分层方式做通，再补持久化、恢复和增强能力；所有实现都服务于“本地优先、任务可恢复、UI 不阻塞”这三个目标。
